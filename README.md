# CS-255-Module 8

1. Briefly summarize the DriverPass project. Who was the client? What type of system did they want to design?
DriverPass is a company seeking to improve student success rates on DMV driving exams by offering structured online learning, practice tests, and on‑the‑road training. The client wanted a cloud‑based system that would allow customers to register, purchase training packages, schedule driving lessons, take online practice tests, and track their progress. The system also needed administrative tools for managing users, vehicles, drivers, appointments, and DMV content updates.

2. What was done particularly well?
The requirements gathering and translation into clear, structured system components were done especially well. The business requirements document accurately captured the client’s needs, including functional requirements, nonfunctional requirements, user roles, and security expectations. The system design documents, the use case diagrams, activity diagrams, class diagrams, and sequence diagrams, aligned tightly with the requirements and demonstrated a strong understanding of how the system should behave from both a user and technical perspective.

3. If you could revise one part of the work, what would it be and how would you improve it?
If I were to revise one part, I would refine the activity diagrams to make the flows even more explicit, especially around scheduling and DMV update processes. Adding clearer branching logic and exception handling would improve readability and better reflect real‑world scenarios, such as handling unavailable drivers or invalid payment information. This would help strengthen the system’s robustness and make the diagrams more useful for developers.

4. How did you interpret the user’s needs and implement them into the system design? Why is it important to consider user needs when designing?
I interpreted the user’s needs by closely analyzing the interview transcript, identifying explicit requirements, and inferring implied needs such as security, usability, and scalability. Each requirement was mapped to system functions, user roles, and design elements to ensure the final system addressed the client’s goals. Considering user needs is essential because a system that does not align with real workflows, expectations, or constraints will fail to solve the client’s problem. Good design ensures the system is intuitive, reliable, and capable of supporting the business long‑term.

5. How do you approach designing software? What techniques or strategies will you use in the future?
I approach software design by starting with clear requirements gathering, followed by modeling the system from both process and object perspectives. I strongly understand visually, which I relly with use case diagrams to understand user interactions, activity diagrams to map workflows, and class diagrams to define system structure. In the future, I will continue using iterative refinement, and modular design to ensure systems are scalable, maintainable, and aligned with user needs. Combining structured analysis with user‑centered thinking leads to stronger, more effective system designs.
